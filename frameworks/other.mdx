bryel is OpenTelemetry-native: anything that emits OpenTelemetry GenAI spans ships to bryel as-is, with no bryel-specific adapter.

## Go through the Vercel AI SDK

The Vercel AI SDK wraps OpenAI, Anthropic, Google, and dozens of other providers behind one interface, and bryel ships a first-class adapter for it. If you can call your model through the AI SDK, it's the least-effort path: models, tokens, cost, and the full tool trajectory, out of the box.

<CodeGroup>

```bash npm
npm i @bryel/vercel @vercel/otel
```

```bash pnpm
pnpm add @bryel/vercel @vercel/otel
```

```bash yarn
yarn add @bryel/vercel @vercel/otel
```

</CodeGroup>

## Call a provider SDK directly

If you call the OpenAI, Anthropic, or LangChain SDKs directly, add any OpenTelemetry instrumentation that emits GenAI spans, then ship those spans with [`bryelSpanProcessor`](/frameworks/opentelemetry). It forwards OTel spans to bryel without modifying them.

<Steps>
  <Step title="Add a GenAI instrumentation">
    Pick the OpenTelemetry instrumentation for your provider SDK. Any instrumentation that emits OpenInference- or OTel-GenAI-shaped spans works.
  </Step>
  <Step title="Register the bryel span processor">
    Wire the instrumentation into the Node SDK alongside `bryelSpanProcessor`.

    ```ts
    import { NodeSDK } from "@opentelemetry/sdk-node";
    import { bryelSpanProcessor } from "@bryel/sdk-core";
    // import the OpenTelemetry instrumentation for your provider SDK

    new NodeSDK({
      instrumentations: [
        /* your GenAI instrumentation here */
      ],
      spanProcessors: [bryelSpanProcessor({ apiKey: process.env.BRYEL_KEY })],
    }).start();
    ```
  </Step>
  <Step title="Run your app">
    Your provider calls now flow to bryel, with no call-site changes.
  </Step>
</Steps>

<Note>
  bryel speaks the OpenInference convention over OTLP and [owns its conventions outright](/api-reference/semconv) — there's no third-party observability SDK in your dependency tree.
</Note>
